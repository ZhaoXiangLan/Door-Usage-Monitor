# Data Models Specification

## Input Model (POST /api/data)

### Description
Used when ESP32 sends door event data to the backend.

### JSON Format

```json
{
  "state": "open",
  "device": "Door_1"
}
```

### Validation Rules

| Field   | Type   | Required | Constraints |
|--------|--------|----------|-------------|
| state  | string | Yes      | Must be `open` |
| device | string | Yes      | Defaults to `"Door_number"` |

---

## MongoDB Collections

### Collection: `door-usage-data`

```javascript
{
    _id: ObjectId,
    time: String,
    state: String,
    device: String
}
```

### Description
Each document represents a single door event.  
The timestamp is generated by the server.

---

## API Response Model (GET /api/data)

### JSON Format

```json
{
  "raw_data": [
    {
      "time": "2026-04-20 14:32:10",
      "state": "open",
      "device": "Door_1"
    }
  ],
  "hourly_data": [
    {
      "hour": "14",
      "count": 5
    }
  ]
}
```

---

## Aggregated Data Model

### hourly_data

| Field | Type   | Description |
|------|--------|-------------|
| hour | string | Hour of the day (00–23) |
| count| integer| Number of events in that hour |

---

## Error Handling Model

### Description
Errors are returned when validation or authentication fails.

### Example

```json
{
  "detail": "Unauthorized"
}
```

```json
{
  "detail": "Missing state"
}
```

---

## Notes

- The `_id` field from MongoDB is excluded from API responses
- Timestamps are stored as formatted strings
- Validation is implemented manually in the backend

---

## Future Improvement

The `time` field is currently stored as a string.  
In future versions, it will be converted to a MongoDB `datetime` type to support TTL-based automatic data cleanup.
